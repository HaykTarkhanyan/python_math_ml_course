\begin{frame}{Why care? Autoencoders are everywhere}
  \small
  Before we build them: the ideas in this short chapter are load-bearing in today's biggest models.
  \vskip 3pt
  \begin{itemize}
    \item \textbf{Self-supervised pretraining.} \textbf{BERT} masks words and rebuilds them;
      \textbf{MAE} masks image patches - "masked modeling" is an autoencoder idea (denoising), and
      the recipe behind most foundation models.
    \item \textbf{LLM interpretability.} A wide \textbf{sparse autoencoder} on a model's activations
      splits them into single-concept features - how Anthropic reads what is inside Claude
      (\emph{Scaling Monosemanticity}, 2024).
    \item \textbf{Cheaper inference.} \textbf{DeepSeek}'s Multi-head Latent Attention squeezes the KV
      cache into a small \textbf{latent vector} ($\sim$4\% of the size), so long contexts fit in
      memory (DeepSeek-V3, 2024).
    \item \textbf{Image generation.} The \textbf{VAE} (next lecture) is the compression layer inside
      Stable Diffusion and Midjourney.
  \end{itemize}
  \vskip 3pt
  \fcolorbox{paramgreen}{paramgreen!10}{\parbox{0.98\linewidth}{\small
    Encoder, bottleneck, denoising, sparse - we build each piece below; watch for where it powers
    the models above.}}

\section{The autoencoder idea}

\begin{frame}[plain]
  \vfill
  \begin{center}
    {\Large\color{popblue}\textbf{The autoencoder idea}}\\[0.9em]
    \begin{minipage}{0.82\textwidth}\centering\small
      Learn to rebuild your input, and let the bottleneck do the teaching.
    \end{minipage}
  \end{center}
  \vfill
\end{frame}

\begin{frame}{Encoder, code, decoder}
  \begin{columns}[T]
    \column{0.48\textwidth}
    \small
    An \textbf{autoencoder} is two networks trained together:
    \vskip 3pt
    \begin{itemize}
      \item \textbf{Encoder} $f:\ \xv \mapsto \mathbf{z}$ - compress the input into a small
        \textbf{code} $\mathbf{z}$.
      \item \textbf{Decoder} $g:\ \mathbf{z} \mapsto \hat{\xv}$ - rebuild the input from that code.
    \end{itemize}
    \vskip 3pt
    Train so $\hat{\xv} \approx \xv$; the target is the input itself, so it needs \textbf{no
    labels} (self-supervised).
    \vskip 5pt
    \fcolorbox{armblue}{armblue!8}{\parbox{0.96\linewidth}{\small
      \textbf{Code / latent.} The code $\mathbf{z}$ is a compact summary of the input; the space
      it lives in is the \textbf{latent space} ("latent" $=$ hidden / underlying). We call
      $\mathbf{z}$ a \emph{code} because the encoder \textbf{encodes} $\xv$ into it - a short
      "password" that stands for the whole image.}}
    \column{0.52\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex]
        \fill[armblue!12] (-2.6,-1.3) -- (-2.6,1.3) -- (-0.5,0.45) -- (-0.5,-0.45) -- cycle;
        \fill[paramgreen!18] (-0.5,-0.45) rectangle (0.5,0.45);
        \fill[armorange!16] (2.6,-1.3) -- (2.6,1.3) -- (0.5,0.45) -- (0.5,-0.45) -- cycle;
        \draw (-2.6,-1.3) -- (-2.6,1.3) -- (-0.5,0.45) -- (-0.5,-0.45) -- cycle;
        \draw (-0.5,-0.45) rectangle (0.5,0.45);
        \draw (2.6,-1.3) -- (2.6,1.3) -- (0.5,0.45) -- (0.5,-0.45) -- cycle;
        \node at (-1.55,0) {encoder $f$};
        \node[align=center] at (0,0) {code\\ $\mathbf{z}$};
        \node at (1.55,0) {decoder $g$};
        \node[below] at (-2.6,-1.35) {$\xv$ (784)};
        \node[below] at (2.6,-1.35) {$\hat{\xv}$ (784)};
        \node[paramgreen, below] at (0,-0.5) {\tiny 2};
      \end{tikzpicture}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{The bottleneck is the whole point}
  \begin{columns}[T]
    \column{0.52\textwidth}
    \small
    Make the code \textbf{smaller} than the input: $\dim(\mathbf{z}) < \dim(\xv)$
    (\textbf{undercomplete}). The network cannot pass everything through, so it must keep
    only what matters - the structure the digits share.
    \vskip 6pt
    \fcolorbox{armred}{armred!8}{\parbox{0.95\linewidth}{\small
      \textbf{No bottleneck, no learning.} If $\dim(\mathbf{z}) \geq \dim(\xv)$ the network
      can just learn the \textbf{identity} - a perfect, useless copy that memorized nothing.}}
    \vskip 6pt
    The \textbf{loss} is nothing new: squared error $\lVert \xv - \hat{\xv}\rVert^2$
    (or cross-entropy for $[0,1]$ pixels). \emph{Callback:} the L2 loss from regression.
    \column{0.48\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex]
        \foreach \y in {-1.4,-1.0,...,1.4} \node[circle, draw, fill=armblue!12,
          inner sep=1.2pt] at (0,\y) {};
        \foreach \y in {-0.4,0,0.4} \node[circle, draw, fill=paramgreen!30,
          inner sep=1.6pt] at (2.2,\y) {};
        \foreach \y in {-1.4,-1.0,...,1.4} \node[circle, draw, fill=armorange!16,
          inner sep=1.2pt] at (4.4,\y) {};
        \node[below] at (0,-1.7) {input};
        \node[paramgreen, below] at (2.2,-1.7) {bottleneck};
        \node[below] at (4.4,-1.7) {output};
      \end{tikzpicture}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{How an autoencoder learns}
  \begin{columns}[T]
    \column{0.48\textwidth}
    \small
    It is the same training loop as any net (ch5, L15) - only the \emph{target} is unusual:
    \vskip 3pt
    \begin{enumerate}
      \item \textbf{Forward:} encode $\mathbf{z}=f(\xv)$, then decode $\hat{\xv}=g(\mathbf{z})$.
      \item \textbf{Loss:} compare the output to the \textbf{input itself},
        $L=\lVert \xv-\hat{\xv}\rVert^2$.
      \item \textbf{Backward:} backprop $L$ through decoder \emph{and} encoder.
      \item \textbf{Update:} one Adam step; repeat over mini-batches.
    \end{enumerate}
    \vskip 3pt
    \fcolorbox{armblue}{armblue!8}{\parbox{0.95\linewidth}{\small
      The one twist vs a normal net: the label \emph{is} the input, so there is nothing to
      hand-label. Everything else - forward, backprop, Adam - is exactly ch5.}}
    \column{0.52\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex,
          b/.style={draw, rounded corners, minimum height=0.6cm, minimum width=1.15cm, align=center}]
        \node[b, fill=armblue!12] (x) at (0,0) {$\xv$};
        \node[b, fill=gray!10] (e) at (1.4,0) {encoder};
        \node[b, fill=paramgreen!22] (z) at (2.8,0) {$\mathbf{z}$};
        \node[b, fill=gray!10] (d) at (4.2,0) {decoder};
        \node[b, fill=armorange!16] (xh) at (5.6,0) {$\hat{\xv}$};
        \draw[->] (x)--(e); \draw[->] (e)--(z); \draw[->] (z)--(d); \draw[->] (d)--(xh);
        \node[b, fill=armred!12, minimum width=2.4cm] (loss) at (2.8,-1.6)
          {loss $=\lVert \xv-\hat{\xv}\rVert^2$};
        \draw[->] (xh) |- (loss);
        \draw[->] (x) |- (loss);
        \draw[->, armred, dashed] (loss) -- (z) node[midway, right] {\tiny backprop};
      \end{tikzpicture}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{Backprop, in symbols}
  \small
  $\hat{\xv} = g\big(f(\xv)\big)$ is a \textbf{composition}, so the chain rule sends the error
  $L=\lVert \xv-\hat{\xv}\rVert^2$ back through \textbf{both} halves ($\theta_f,\theta_g$ = encoder
  / decoder weights):
  \[
    \underbrace{\frac{\partial L}{\partial \theta_g}}_{\text{decoder}}
      = \frac{\partial L}{\partial \hat{\xv}}\,\frac{\partial \hat{\xv}}{\partial \theta_g},
    \qquad
    \underbrace{\frac{\partial L}{\partial \theta_f}}_{\text{encoder}}
      = \underbrace{\frac{\partial L}{\partial \hat{\xv}}\,\frac{\partial \hat{\xv}}{\partial \mathbf{z}}}_{\text{back through the decoder}}\,
        \frac{\partial \mathbf{z}}{\partial \theta_f}.
  \]
  \vskip -3pt
  \begin{center}
    \begin{tikzpicture}[font=\scriptsize, >=Latex,
        b/.style={draw, rounded corners, minimum height=0.5cm, minimum width=1.02cm}]
      \node[b, fill=armblue!12] (x) at (0,0) {$\xv$};
      \node[b, fill=gray!12] (e) at (1.5,0) {encoder};
      \node[b, fill=paramgreen!22] (z) at (3.0,0) {$\mathbf{z}$};
      \node[b, fill=gray!12] (d) at (4.5,0) {decoder};
      \node[b, fill=armorange!16] (xh) at (6.0,0) {$\hat{\xv}$};
      \node (L) at (7.1,0) {$L$};
      \draw[->] (x)--(e); \draw[->] (e)--(z); \draw[->] (z)--(d); \draw[->] (d)--(xh); \draw[->] (xh)--(L);
      \draw[->, armred, thick] (L) to[bend right=42] (d);
      \draw[->, armred, thick] (d) to[bend right=42] (z);
      \draw[->, armred, thick] (z) to[bend right=42] (e);
      \node[armred] at (3.55,-0.95) {\tiny gradient flows back: through the decoder $\to$ the code $\to$ the encoder};
    \end{tikzpicture}
  \end{center}
  \vskip -1pt
  \fcolorbox{armblue}{armblue!8}{\parbox{0.98\linewidth}{\small
    The \textbf{encoder} is trained \emph{through} the \textbf{decoder}: the decoder first says
    "this code would rebuild $\xv$ better", then the encoder learns to produce such codes - which
    is why the two halves share one representation. \texttt{loss.backward()} does it all.}}
\end{frame}

\begin{frame}{You already built the decoder}
  \begin{columns}[T]
    \column{0.46\textwidth}
    \small
    For images, the two halves are just a \textbf{CNN and its mirror} - a
    \textbf{convolutional autoencoder}:
    \vskip 4pt
    \begin{itemize}
      \item Encoder = convolution + pooling (\emph{ch6, L16}) - shrink to a code.
      \item Decoder = \textbf{transposed convolution} (\emph{L19}) - grow the code back
        to an image.
    \end{itemize}
    \vskip 6pt
    \fcolorbox{armblue}{armblue!8}{\parbox{0.95\linewidth}{\small
      Nothing new to learn here: you built both halves in the CNN chapter. An autoencoder
      just points them at each other.}}
    \column{0.54\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex]
        \draw[fill=armblue!12] (-2.8,-1.2) -- (-2.8,1.2) -- (-0.4,0.35) -- (-0.4,-0.35) -- cycle;
        \draw[fill=armorange!16] (2.8,-1.2) -- (2.8,1.2) -- (0.4,0.35) -- (0.4,-0.35) -- cycle;
        \draw[fill=paramgreen!25] (-0.4,-0.35) rectangle (0.4,0.35);
        \node[rotate=90] at (-1.6,0) {conv + pool};
        \node[rotate=90] at (1.6,0) {transposed conv};
        \node[paramgreen] at (0,0) {\tiny code};
      \end{tikzpicture}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}[fragile]{In practice (PyTorch)}
  \small
  A working autoencoder is a handful of lines - encoder, decoder, and the copy-yourself loop:
  \vskip 2pt
  \begin{columns}[T]
    \column{0.54\textwidth}
    \scriptsize
\begin{verbatim}
enc = nn.Sequential(
  nn.Linear(784,256), nn.ReLU(),
  nn.Linear(256,32))          # code
dec = nn.Sequential(
  nn.Linear(32,256), nn.ReLU(),
  nn.Linear(256,784), nn.Sigmoid())

for xb in loader:             # no labels!
  xh = dec(enc(xb))
  loss = F.mse_loss(xh, xb)
  opt.zero_grad()
  loss.backward(); opt.step()
\end{verbatim}
    \column{0.46\textwidth}
    \textbf{Practical notes}
    \begin{itemize}
      \item Scale pixels to $[0,1]$; use \texttt{mse\_loss} (or \texttt{BCE} with the
        \texttt{Sigmoid} output).
      \item The code size is your main knob - start around \textbf{16-64}.
      \item \textbf{Identity trap}: perfect reconstruction but a useless code $\Rightarrow$ the
        code is too big. Shrink it, or add noise / an $L_1$ penalty.
      \item For images, swap \texttt{Linear} for \texttt{Conv2d} / \texttt{ConvTranspose2d}.
    \end{itemize}
  \end{columns}
\end{frame}

\section{Autoencoders = nonlinear PCA}

\begin{frame}[plain]
  \vfill
  \begin{center}
    {\Large\color{popblue}\textbf{Autoencoders = nonlinear PCA}}\\[0.9em]
    \begin{minipage}{0.82\textwidth}\centering\small
      You already compressed digits once - PCA. This bends the axes.
    \end{minipage}
  \end{center}
  \vfill
\end{frame}

\begin{frame}{PCA is already an encoder-decoder}
  \begin{columns}[T]
    \column{0.54\textwidth}
    \small
    Rewind to \textbf{PCA} (ch4b). To compress a digit it does two steps:
    \vskip 3pt
    \begin{itemize}
      \item \textbf{Encode:} project the 784 pixels onto the top $k$ directions $\to$ $k$ numbers.
      \item \textbf{Decode:} multiply those $k$ numbers back out $\to$ a reconstruction
        $\hat{\xv}$.
    \end{itemize}
    \vskip 3pt
    That is \emph{exactly} an autoencoder - only with \textbf{linear} encode and decode and no
    hidden layers.
    \vskip 5pt
    \fcolorbox{armblue}{armblue!8}{\parbox{0.95\linewidth}{\small
      So PCA is the \textbf{simplest possible autoencoder}. The real question: what changes when
      the encoder and decoder become real (nonlinear) networks?}}
    \column{0.46\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex,
          b/.style={draw, rounded corners, align=center}]
        \node[b, fill=armblue!10, minimum width=1.3cm, minimum height=1.3cm] (x) at (0,0)
          {$\xv$\\ \tiny 784};
        \node[b, fill=paramgreen!22, minimum width=0.8cm, minimum height=0.7cm] (z) at (2.3,0) {$k$};
        \node[b, fill=armorange!14, minimum width=1.3cm, minimum height=1.3cm] (xh) at (4.4,0)
          {$\hat{\xv}$\\ \tiny 784};
        \draw[->] (x) -- (z) node[midway, above] {\tiny project};
        \draw[->] (z) -- (xh) node[midway, above] {\tiny expand};
      \end{tikzpicture}
      \vskip 4pt
      {\scriptsize both arrows are just matrix multiplies}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{From PCA to a real autoencoder}
  \small
  Keep the encoder and decoder \textbf{linear} - the activation is the \emph{identity}
  $\phi(a)=a$ (just weighted sums, no bend) - and minimize reconstruction error: the autoencoder
  provably recovers the \textbf{same subspace as PCA} - the best flat sheet through the data
  (directions of maximum variance).
  \vskip 4pt
  Now switch the activations \textbf{on} (ReLU, sigmoid). Encode and decode become
  \textbf{curved}, so the code can follow structure a flat sheet cannot. That is the whole idea:
  \vskip 6pt
  \begin{center}
    \fcolorbox{popblue}{popblue!6}{\parbox{0.92\linewidth}{\centering
      \textbf{linear} autoencoder $=$ PCA \qquad$\longrightarrow$\qquad
      \textbf{nonlinear} autoencoder $=$ PCA that can bend}}
  \end{center}
  \vskip 6pt
  \fcolorbox{armblue}{armblue!8}{\parbox{0.98\linewidth}{\small
    You do not throw PCA away - you \textbf{generalize} it. The next two slides show the
    difference: first on a curved toy dataset, then on the digits.}}
\end{frame}

\begin{frame}{See it: PCA flattens, the autoencoder bends}
  \small
  A \textbf{swiss roll} - a 2-D sheet rolled up in 3-D. PCA can only take a flat photo of it; a
  nonlinear method unrolls it back to the sheet:
  \begin{center}
    \includegraphics[width=0.99\linewidth]{manifold_pca_vs_ae.pdf}
  \end{center}
  \fcolorbox{armblue}{armblue!8}{\parbox{0.99\linewidth}{\small
    In PCA's flat photo (middle) far-apart points on the roll \textbf{land on top of each other};
    the nonlinear map (right; Isomap here, an autoencoder does the same) \textbf{unrolls} it so the
    colour runs smoothly. That is what "bend" buys you.}}
\end{frame}

\begin{frame}{Back to digits: the 2-number experiment}
  \begin{columns}[T]
    \column{0.54\textwidth}
    \small
    Same task, head to head. Force each test digit through a \textbf{2-number} bottleneck, then
    rebuild it:
    \vskip 3pt
    \begin{itemize}
      \item \textbf{PCA}: keep only the top \textbf{2 components}, reconstruct (linear).
      \item \textbf{Autoencoder}: a $784\!\to\!2\!\to\!784$ net with nonlinear activations.
    \end{itemize}
    \vskip 3pt
    Both get the \emph{same} 2-number budget; the only difference is linear vs curved.
    \vskip 5pt
    \fcolorbox{armorange}{armorange!12}{\parbox{0.95\linewidth}{\small
      2 numbers is a brutal squeeze for a 784-pixel digit - neither will be perfect. The question
      is \emph{which structure survives}.}}
    \column{0.46\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex, b/.style={draw, rounded corners, align=center}]
        \node[b, fill=armblue!10, minimum width=1.1cm, minimum height=1.1cm] (x) at (0,0)
          {digit\\ \tiny 784};
        \node[b, fill=paramgreen!25, minimum width=0.65cm, minimum height=0.6cm] (z) at (2.0,0) {2};
        \node[b, fill=armorange!14, minimum width=1.1cm, minimum height=1.1cm] (xh) at (3.9,0)
          {rebuilt\\ \tiny 784};
        \draw[->] (x) -- (z); \draw[->] (z) -- (xh);
        \node[below] at (2.0,-0.45) {\tiny squeeze};
      \end{tikzpicture}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{The result: nonlinear keeps more}
  \begin{center}
    \includegraphics[width=0.96\linewidth]{recon_pca_vs_ae.pdf}
  \end{center}
  \fcolorbox{armorange}{armorange!12}{\parbox{0.98\linewidth}{\small
    PCA's 2-component reconstruction (middle row) collapses several digits to a ghostly

\begin{frame}{How big should the code be?}
  \small
  The code size - the \textbf{latent dimension} - is the key knob (we used 2 only so we could
  \emph{draw} the space):
  \begin{center}
    \includegraphics[width=0.68\linewidth]{latent_dim_recon.pdf}
  \end{center}
  \vskip -3pt
  \fcolorbox{armblue}{armblue!8}{\parbox{0.98\linewidth}{\small
    Too small (2): the net \textbf{guesses} (a 5 comes back a 9). Bigger (8, 32): sharper, classes
    separate - but you lose the 2-D picture. Real autoencoders use \textbf{tens to hundreds}.}}
\end{frame}

\begin{frame}{The same digits as a 2-D map}
  \begin{columns}[T]
    \column{0.34\textwidth}
    \small
    Both methods also give a \textbf{2-D map} of the data (each point is one digit).
    \vskip 6pt
    \fcolorbox{armorange}{armorange!12}{\parbox{0.95\linewidth}{\small
      Honest caveat: the AE's axes are \textbf{not} orthogonal, \textbf{not} variance-ordered,
      and \textbf{not} unique (another training run gives a different map). PCA hands you all
      of those for free. The AE trades them for flexibility.}}
    \column{0.66\textwidth}
    \begin{center}
      \includegraphics[width=\linewidth]{pca_vs_ae_mnist.pdf}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{The latent captures more than pixels}
  \small
  That 2-D map is not just pretty - the code captures \textbf{structure you can use}.
  \vskip 4pt
  A striking example from medicine: a patient's \textbf{sex} is nearly impossible to read off a raw
  \textbf{brain MRI} - even trained radiologists cannot do it by eye. Yet train an autoencoder on
  the scans, look at the latent, and \textbf{men and women separate}.
  \vskip 6pt
  \fcolorbox{armblue}{armblue!8}{\parbox{0.98\linewidth}{\small
    The autoencoder surfaced a signal that \emph{was} in the data but hidden from the eye - here
    enough to fill a missing field in a patient record. That is \textbf{representation learning}:
    a compact code that makes hidden structure explicit. \emph{(Illustrative - a known result on
    brain imaging, not run here.)}}}
\end{frame}

\section{Regularized autoencoders}

\begin{frame}[plain]
  \vfill
  \begin{center}
    {\Large\color{popblue}\textbf{Regularized autoencoders}}\\[0.9em]
    \begin{minipage}{0.82\textwidth}\centering\small
      When you cannot rely on a tight bottleneck, constrain the code another way.
    \end{minipage}
  \end{center}
  \vfill
\end{frame}

\begin{frame}{Why go overcomplete?}
  \begin{columns}[T]
    \column{0.56\textwidth}
    \small
    Compression is not always the goal. Sometimes you want to \textbf{re-represent} the input as a
    \emph{richer} set of features - a code \textbf{bigger} than the input, called
    \textbf{overcomplete}. Two reasons it helps:
    \vskip 4pt
    \begin{itemize}
      \item \textbf{A vocabulary, not a summary.} A language has thousands of words, but any one
        sentence uses a handful. An overcomplete code is a big \textbf{dictionary} of
        feature-detectors; each image switches on just the \textbf{few} it needs (like the brain:
        many neurons, few firing at once).
      \item \textbf{Spread out to separate.} Structure tangled in the raw pixels can become easy to
        read once the input is lifted into a bigger, richer space.
    \end{itemize}
    \column{0.44\textwidth}
    \begin{center}
      \begin{tikzpicture}[font=\scriptsize, >=Latex]
        \foreach \y in {-1.2,-0.8,...,1.2} \node[circle, draw, fill=armblue!12,
          inner sep=1.1pt] at (0,\y) {};
        \foreach \y in {-1.4,-1.0,...,1.4} \node[circle, draw, fill=paramgreen!20,
          inner sep=1.3pt] at (2.0,\y) {};
        \foreach \y in {-1.2,-0.8,...,1.2} \node[circle, draw, fill=armorange!16,
          inner sep=1.1pt] at (4.0,\y) {};
        \node[below] at (0,-1.7) {input};
        \node[paramgreen, below] at (2.0,-1.7) {\emph{bigger} code};
        \node[below] at (4.0,-1.7) {output};
        \node[armred] at (2.0,2.0) {no bottleneck!};
      \end{tikzpicture}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{...but a big code can cheat}
  \small
  An overcomplete code has room to spare - so the network can take the lazy way out and just
  \textbf{copy the input straight through} (the identity map again), learning nothing useful.
  \vskip 8pt
  \fcolorbox{armred}{armred!8}{\parbox{0.98\linewidth}{\small
    With a big code we can no longer lean on a tight bottleneck to force learning. We need a
    \emph{different} constraint to keep the network honest. Three classic fixes, coming up:
    \textbf{denoising} (corrupt the input), \textbf{sparse} (penalize how much the code fires),
    and \textbf{contractive} (penalize the code's sensitivity to the input).}}
\end{frame}

\begin{frame}{Denoising autoencoders}
  \begin{columns}[T]
    \column{0.34\textwidth}
    \small
    \textbf{Break the input on purpose.} Corrupt $\xv$ (add noise, or zero out pixels),
    then train the network to output the \textbf{clean} original.
    \vskip 4pt
    It cannot copy noise, so it must learn what a digit really is.
    \vskip 6pt
    \fcolorbox{armblue}{armblue!8}{\parbox{0.95\linewidth}{\small
      \emph{Callback:} ch5 data augmentation - robustness by perturbation. Here the
      perturbation is the whole training signal.}}
    \vskip 4pt
    {\footnotesize \textbf{Predict:} zero out half the pixels and demand the clean digit
    back - does that really help? \emph{Yes.}}
    \column{0.66\textwidth}
    \begin{center}
      \includegraphics[width=\linewidth]{denoising_grid.pdf}
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}{Sparse autoencoders}
  \small
  Constraint \#2: don't shrink the code, \textbf{penalize how much it fires}. Add an $L_1$ penalty
  on the code activations (or a target average activation), so for any one input only a
  \textbf{few} of the many code neurons switch on.
  \begin{center}
    \includegraphics[width=0.9\linewidth]{sparse_activations.pdf}
  \end{center}
  \fcolorbox{armblue}{armblue!8}{\parbox{0.98\linewidth}{\small
    Each input now lights up a \textbf{small, specialised} set of detectors - more interpretable,
    and it \emph{cannot} copy the input (that would need every neuron). \emph{Callback:} the $L_1$
    penalty is Lasso (ch1), now on \textbf{activations} instead of weights.}}
\end{frame}

\begin{frame}{Contractive autoencoders}
  \small
  Constraint \#3: make the code \textbf{insensitive to tiny input wiggles}. Penalize how fast the
  encoder $f$ reacts as the input moves - the squared Frobenius norm of its Jacobian:
  \[
    L\big(\xv,\, g(f(\xv))\big) \;+\; \lambda \left\| \frac{\partial f(\xv)}{\partial \xv} \right\|_F^2 .
  \]
  Now two nearby inputs map to \textbf{almost the same code}: the encoder contracts small
  perturbations away and keeps only the few directions that actually matter - those along the
  data manifold.
  \vskip 6pt
  \fcolorbox{armblue}{armblue!8}{\parbox{0.98\linewidth}{\small
    \emph{Denoising vs contractive:} the DAE earns a stable code by \textbf{corrupting inputs}
    (sampled noise); the CAE asks for it \textbf{directly}, through a deterministic Jacobian
    penalty. Same goal, two routes. \emph{Callback:} it is just another term added to the loss -
    exactly like weight decay in ch5.}}
\end{frame}

\section{Anomaly detection}

\begin{frame}[plain]
  \vfill
  \begin{center}
    {\Large\color{popblue}\textbf{Anomaly detection}}\\[0.9em]
    \begin{minipage}{0.82\textwidth}\centering\small
      If it cannot rebuild you, you do not belong.
    \end{minipage}
  \end{center}
  \vfill
\end{frame}

\begin{frame}{Reconstruction error is an anomaly score}
  \begin{columns}[T]
    \column{0.44\textwidth}
    \small
    Train an autoencoder on \textbf{normal data only}. It learns to rebuild normal inputs
    with \textbf{low} error. Feed it something it has never seen - it rebuilds it
    \textbf{badly}, with \textbf{high} error.
    \vskip 4pt
    So the reconstruction error \emph{is} an anomaly score: pick a \textbf{threshold},
    flag anything above it.
    \vskip 6pt
    \fcolorbox{armblue}{armblue!8}{\parbox{0.95\linewidth}{\small
      The industrial workhorse use of autoencoders: fraud, manufacturing defects, network
      intrusion. \emph{Callback:} thresholding a score - ch3 metrics.}}
    \column{0.56\textwidth}
    \begin{center}
      \includegraphics[width=\linewidth]{anomaly_recon_error_hist.pdf}
    \end{center}
    {\scriptsize Trained only on 1s; a 0 gets forced toward a 1-shape $\Rightarrow$ high error.}
  \end{columns}
\end{frame}

\begin{frame}{A caveat you can see}
  \small
  Reconstruction-error anomaly detection only works when the anomaly is \textbf{genuinely
  different} from the training data.
  \vskip 6pt
  \fcolorbox{armorange}{armorange!12}{\parbox{0.98\linewidth}{\small
    Train on digits 0-8 and try to flag a \textbf{9}: it fails - a 9 looks like a 4, a 7,
    a 1, all of which the network saw, so it reconstructs 9s just fine. The clean split on
    the previous slide needed a truly distinct anomaly (train on 1s, flag 0s). Know what
    "normal" and "anomalous" mean for \emph{your} data before you trust the error.}}
\end{frame}

\section*{Recap}

\begin{frame}{Recap: autoencoders}
  \small
  \begin{itemize}
    \item An autoencoder is an \textbf{encoder + decoder} trained to rebuild its input -
      \textbf{self-supervised}, no labels.
    \item The \textbf{bottleneck} forces it to keep only structure; a linear one recovers
      \textbf{PCA}, a nonlinear one bends around curved data.
    \item Cannot rely on a tight bottleneck? Regularize: \textbf{denoising} (corrupt and
      recover), \textbf{sparse} (penalize activations), or \textbf{contractive} (penalize
      code sensitivity).
    \item \textbf{Anomaly detection}: high reconstruction error flags what the network
      never learned.
  \end{itemize}
\end{frame}

\begin{frame}{But it cannot generate}
  \begin{columns}[T]
    \column{0.40\textwidth}
    \small
    One thing the autoencoder \textbf{cannot} do: invent a new digit. To generate, you
    would draw a random code $\mathbf{z}$ and decode it. But the AE never told its codes to
    follow any known distribution.
    \vskip 6pt
    \fcolorbox{armred}{armred!8}{\parbox{0.95\linewidth}{\small
      Draw $\mathbf{z}\sim\mathcal{N}(0,I)$ (red) and the codes (blue) are somewhere else
      entirely: the draws miss almost the whole manifold and decode to blurry blobs. You
      can \textbf{reconstruct}, but you cannot \textbf{sample}.}}
    \vskip 6pt
    \fcolorbox{paramgreen}{paramgreen!10}{\parbox{0.95\linewidth}{\small
      \textbf{Next (L23):} give the latent space no holes, then sample it -
      \textbf{variational autoencoders}.}}
    \column{0.60\textwidth}
    \begin{center}
      \includegraphics[width=\linewidth]{ae_sampling_fail.pdf}
    \end{center}
  \end{columns}
\end{frame}
