\section{Missing data handling}

\begin{frame}[fragile]{The problem with \texttt{NaN}}
  Real datasets have holes. A single \texttt{NaN} breaks math operations downstream.
  \vfill
  Running example: Yerevan apartment rent. Some rows have unknown \texttt{balcony}, others have unknown \texttt{year\_built}.
  \vfill
\begin{lstlisting}
>>> df.head()
   area  rooms  district     year_built  balcony  rent
0  45.0    2    Kentron       1985        Yes     280
1  62.0    3    Arabkir         NaN        No     340
2   NaN    2    Komitas       1992        NaN     310
3  38.0    1    Kentron       1978        No     245

>>> (df["area"] * 1000).mean()
nan                        # one NaN poisons the whole column
\end{lstlisting}
  \vfill
  Before any model: detect, decide, document.
\end{frame}

\begin{frame}{Types of missingness}
  \emph{Why} data is missing matters more than how much. Plain-English first, statistics names in parentheses:
  \vfill
  \begin{itemize}
    \item \textbf{Innocent missing (MCAR).} The reason a value is missing has nothing to do with any other variable. Example: random data-entry hiccup. \textit{Safe to drop or impute.}
    \item \textbf{Structured missing (MAR).} Missingness depends on \emph{other observed} variables. Example: older buildings less likely to have \texttt{year\_built} recorded - missingness depends on the (observable) age. \textit{A model can recover it from the other columns.}
    \item \textbf{Biased missing (MNAR).} Missingness depends on the \emph{value itself}. Example: tenants don't report rent when it's unusually high - the value affects whether it's reported. \textit{Dangerous - imputation will be biased; need domain reasoning.}
  \end{itemize}
  \vfill
  You can't fully distinguish these from data alone. You can usually rule things out by talking to whoever collected the data.
\end{frame}

\begin{frame}[fragile]{Strategy 1 \& 2: drop vs.\ simple impute}
  \begin{columns}[T]
    \column{0.5\textwidth}
    \textbf{Drop} - cheap, sometimes wrong:
\begin{lstlisting}
df.dropna()              # any NaN
df.dropna(thresh=4)      # keep rows w/ >= 4 non-NaN
df.dropna(axis=1)        # drop NaN columns
\end{lstlisting}
    \begin{itemize}\small
      \item Use when missing rate is small ($< 5\%$) AND MCAR-ish.
      \item Risk: throwing away signal. If \texttt{balcony} is missing for 30\% of rows, dropping them loses a third of the data.
    \end{itemize}
    \column{0.5\textwidth}
    \textbf{Simple impute} - fast baseline:
\begin{lstlisting}
from sklearn.impute import SimpleImputer

num = SimpleImputer(strategy="median")
cat = SimpleImputer(strategy="most_frequent")
df["area"] = num.fit_transform(df[["area"]])
\end{lstlisting}
    \begin{itemize}\small
      \item Numeric: median (robust to outliers), mean (only if symmetric).
      \item Categorical: mode.
      \item Constants: \texttt{fill\_value=0} or \texttt{"Unknown"}.
    \end{itemize}
  \end{columns}
\end{frame}

\begin{frame}{Predict-first: what does mean imputation hide?}
  You have area data for 100 apartments. 20 values are missing. You fill the 20 holes with the mean of the other 80.
  \vfill
  \pause
  \textbf{Predict before reading on:} after imputation,
  \begin{itemize}
    \item What happens to the \emph{variance} of the column?
    \item What happens to its \emph{correlation} with rent?
    \item What happens to a model's reported \emph{confidence}?
  \end{itemize}
  \vfill
  \pause
  \begin{columns}[T]
    \column{0.55\textwidth}
    \small
    \begin{itemize}
      \item \textbf{Variance shrinks} - the 20 imputed values are identical, so they contribute 0 to spread.
      \item \textbf{Correlation attenuates} - 20 rows say ``I am the mean'' regardless of rent.
      \item \textbf{Confidence is overstated} - the model treats imputed values as real measurements.
    \end{itemize}
    Mean imputation is a fine baseline, but it lies about uncertainty.
    \column{0.42\textwidth}
    \begin{center}
      \begin{tikzpicture}

          \addplot[ybar, fill=paramgreen, draw=paramgreen, bar width=4pt] coordinates

        \end{axis}

      \scriptsize Green spike at the mean = lost variance.
    \end{center}
  \end{columns}
\end{frame}

\begin{frame}[fragile]{Smarter imputation + the indicator trick}
  \textbf{Model-based imputation.} Predict each missing value from the other features.
\begin{lstlisting}
from sklearn.impute import KNNImputer, IterativeImputer
KNNImputer(n_neighbors=5).fit_transform(X)
IterativeImputer(random_state=509).fit_transform(X)   # ~MICE
\end{lstlisting}
  \begin{itemize}\small
    \item KNN: find the 5 most similar rows, average their value.
    \item Iterative (MICE-style): regress each column on the others, loop until stable.
    \item Cost: slower, can leak if not fit on train only.
  \end{itemize}
  \vfill
  \textbf{Missingness as a feature.} Add a binary indicator column:
\begin{lstlisting}
df["balcony_missing"] = df["balcony"].isna().astype(int)
df["balcony"] = df["balcony"].fillna("Unknown")
\end{lstlisting}
  Why: \emph{the fact that something is missing} can itself be predictive. A landlord who didn't report \texttt{year\_built} may be hiding a very old building.
\end{frame}

\begin{frame}[fragile]{Two more cleanups: duplicates and outliers}
  \begin{columns}[T]
    \column{0.5\textwidth}
    \textbf{Duplicates.}
{\lstset{basicstyle=\ttfamily\scriptsize}
\begin{lstlisting}
df.duplicated().sum()           # count
df.drop_duplicates()            # exact
df.drop_duplicates(             # subset
    subset=["area","rooms","district"])
\end{lstlisting}}
    \begin{itemize}\small
      \item Exact duplicates: usually safe to drop.
      \item \emph{Near}-duplicates (same apartment listed twice with tiny price diff): need a domain rule.
      \item Sometimes duplicates are real (two units of one model car) - check before dropping.
    \end{itemize}
    \column{0.5\textwidth}
    \textbf{Outliers.}
{\lstset{basicstyle=\ttfamily\scriptsize}
\begin{lstlisting}
# IQR rule (Tukey)
q1, q3 = df["rent"].quantile([.25, .75])
iqr = q3 - q1
mask = df["rent"].between(q1 - 1.5*iqr,
                          q3 + 1.5*iqr)
# z-score rule
from scipy.stats import zscore
mask = abs(zscore(df["rent"])) < 3
# model-based: IsolationForest (sklearn)
\end{lstlisting}}
    \begin{itemize}\small
      \item Don't drop blindly. Investigate first.
      \item Outliers are often data-entry errors OR the most interesting rows.
    \end{itemize}
  \end{columns}
\end{frame}

\section{Categorical variable encoding}

\begin{frame}{Why encode at all?}
  Most ML models (linear regression, k-NN, gradient boosting trees, neural networks) operate on \textbf{numbers}. They can't multiply, take dot products with, or compute distances on the string \texttt{"Kentron"}.
  \vfill
  Encoding turns categorical strings (or category codes) into numerical features that preserve the right structure.
  \vfill
  Pick the encoding by two questions:
  \begin{enumerate}
    \item \textbf{Are the categories ordered?} (e.g., \emph{small} $<$ \emph{medium} $<$ \emph{large}.)
    \item \textbf{How many unique values?} (Few $\to$ one-hot is fine. Many $\to$ one-hot explodes.)
  \end{enumerate}
\end{frame}

\begin{frame}[fragile]{One-hot encoding: the default for unordered}
  Each category becomes its own 0/1 column.
  \vfill
  \begin{columns}[T]
    \column{0.45\textwidth}
    \textbf{Before:}
    \begin{tabular}{l}
      \texttt{district} \\ \hline
      Kentron \\
      Arabkir \\
      Komitas \\
      Kentron
    \end{tabular}
    \column{0.55\textwidth}
    \textbf{After one-hot:}
    \begin{tabular}{ccc}
      \texttt{d=Kentron} & \texttt{d=Arabkir} & \texttt{d=Komitas} \\ \hline
      1 & 0 & 0 \\
      0 & 1 & 0 \\
      0 & 0 & 1 \\
      1 & 0 & 0
    \end{tabular}
  \end{columns}
  \vfill
\begin{lstlisting}
pd.get_dummies(df, columns=["district"], drop_first=False)
from sklearn.preprocessing import OneHotEncoder
OneHotEncoder(handle_unknown="ignore", sparse_output=False)
\end{lstlisting}
  \vfill
  \fcolorbox{armred}{armred!8}{\parbox{0.96\textwidth}{\small
    \textbf{Production essential:} \texttt{handle\_unknown="ignore"}. Otherwise an unseen test-set district crashes the model. \texttt{drop="first"} also matters - next slide.
  }}
\end{frame}

\begin{frame}{The dummy trap (link back to L01b)}
  Including \emph{all} one-hot columns AND an intercept makes the design matrix singular:
  \[
    \underbrace{1}_{\text{intercept}} = \underbrace{\mathds{1}[d{=}\text{Kentron}] + \mathds{1}[d{=}\text{Arabkir}] + \mathds{1}[d{=}\text{Komitas}]}_{\text{sum of all dummies}}
  \]
  The all-ones column equals the sum of the dummies. Columns are linearly dependent $\Rightarrow$ $\Xmat^\top\Xmat$ not invertible $\Rightarrow$ OLS fails (L01b Section 4).
  \vfill
  \textbf{Two fixes:}
  \begin{itemize}
    \item \texttt{pd.get\_dummies(..., drop\_first=True)} - drops one category, which becomes the reference baseline absorbed into $\theta_0$.
    \item \texttt{OneHotEncoder(drop="first")} (sklearn).
  \end{itemize}
  \vfill
  \textbf{Exceptions where you keep ALL dummies:}
  \begin{itemize}\small
    \item Ridge / Lasso (regularization makes the system invertible anyway).
    \item Trees (no intercept, no matrix inversion).
    \item When you want all coefficients interpretable on the same footing (no reference category).
  \end{itemize}
\end{frame}

\begin{frame}[fragile]{Label / ordinal encoding: ONLY for ordered}
  Maps each category to an integer. \textbf{Imposes an order.}
  \vfill
\begin{lstlisting}
from sklearn.preprocessing import OrdinalEncoder
# CORRECT use: t-shirt sizes
OrdinalEncoder(categories=[["S","M","L","XL"]]).fit_transform(...)
# WRONG use: city names
OrdinalEncoder().fit_transform(df[["district"]])  # treats Kentron=0 < Arabkir=1 < Komitas=2
\end{lstlisting}
  \vfill
  \textbf{The common mistake:} feeding label-encoded city names to a linear regression. The model sees Komitas = $2 \times$ Arabkir, which is nonsense.
  \vfill
  Rule: use ordinal encoding only when you can write down the order on paper and defend it.
\end{frame}

\begin{frame}[fragile]{High-cardinality: target and frequency encoding}
  Some categoricals have hundreds or thousands of levels (zip codes, product IDs). One-hot explodes; ordinal lies. Two production tricks:
  \vfill
  \textbf{Target encoding.} Replace each category by the mean target value among rows of that category.
\begin{lstlisting}
# example: district -> mean rent in that district (computed on TRAIN only!)
mean_rent = df_train.groupby("district")["rent"].mean()
df["district_te"] = df["district"].map(mean_rent)
\end{lstlisting}
  \footnotesize\textbf{Why this is leakage if computed on full data:} the mean depends on \texttt{rent}, which is what you're predicting. If you compute the mean using test rows, you've used the answer as an input.\normalsize
  \vfill
  \textbf{Frequency / count encoding.} Replace each category by how often it appears.
\begin{lstlisting}
freq = df_train["district"].value_counts(normalize=True)
df["district_freq"] = df["district"].map(freq)
\end{lstlisting}
  Both must be \emph{fit on the training fold only}. Use \texttt{category\_encoders} library + CV-aware variants in practice.
\end{frame}

\begin{frame}[fragile]{Datetime preprocessing}
  Raw dates aren't directly usable. Extract components, and use \emph{cyclic} encoding for periodic features.
\begin{lstlisting}
df["dt"] = pd.to_datetime(df["listed_at"])
df["year"]    = df["dt"].dt.year
df["month"]   = df["dt"].dt.month
df["weekday"] = df["dt"].dt.dayofweek          # 0=Mon
df["hour"]    = df["dt"].dt.hour
\end{lstlisting}
  \textbf{Cyclic features need sin/cos encoding.} \texttt{hour=23} and \texttt{hour=0} are 1 hour apart, not 23. Label encoding lies; one-hot wastes columns. Trick:
\begin{lstlisting}
import numpy as np
df["hour_sin"] = np.sin(2 * np.pi * df["hour"] / 24)
df["hour_cos"] = np.cos(2 * np.pi * df["hour"] / 24)
\end{lstlisting}
  \vfill
  Also useful: time since a reference event (\texttt{days\_since\_listed}), holiday indicators, weekend flag.
\end{frame}

\section{Scaling and normalization}

\begin{frame}{Predict-first: why scale?}
  You have three features:
  \begin{itemize}
    \item \texttt{area} in m$^2$, ranging 20-200
    \item \texttt{rooms}, ranging 1-5
    \item \texttt{year\_built}, ranging 1950-2024
  \end{itemize}
  \vfill
  \pause
  \textbf{Predict before reading on:}
  \begin{itemize}
    \item In k-NN with Euclidean distance, which feature dominates the neighbor calculation?
    \item In gradient descent on linear regression, does the step size that works for $\theta_{\text{area}}$ also work for $\theta_{\text{year}}$?
    \item Do decision trees care about feature scale at all?
  \end{itemize}
  \vfill
  \pause
  Answers: kNN distance is dominated by \texttt{year\_built} (largest range). GD oscillates: one good learning rate per feature is impossible. Trees: \textbf{no} - they only use threshold comparisons.
\end{frame}

\begin{frame}{Three scalers: formulas + when to use}
  \begin{columns}[T]
    \column{0.32\textwidth}
    \textbf{StandardScaler} (z-score):
    \[
      x' = \frac{x - \mu}{\sigma}
    \]
    Output: mean 0, std 1.
    \vskip 4pt
    \footnotesize Best for \emph{roughly Gaussian} features. Default choice for linear models.
    \column{0.32\textwidth}
    \textbf{MinMaxScaler}:
    \[
      x' = \frac{x - \min}{\max - \min}
    \]
    Output: range $[0, 1]$.
    \vskip 4pt
    \footnotesize Best for \emph{bounded inputs} (image pixels, percentages, anything with natural floor/ceiling).
    \column{0.32\textwidth}
    \textbf{RobustScaler}:
    \[
      x' = \frac{x - \text{med}}{\text{IQR}}
    \]
    Output: median 0, IQR 1.
    \vskip 4pt
    \footnotesize Best when \emph{outliers} would distort mean/std. Uses median + IQR instead.
  \end{columns}
  \vfill
  \begin{itemize}\small
    \item All three are linear ($x' = a\,x + b$). They don't change the \emph{shape} of the distribution.
    \item For a skewed feature: apply log / power transform \emph{first}, then scale.
  \end{itemize}
\end{frame}

\begin{frame}[fragile]{Three scalers: code comparison}
  Same interface for all three - drop-in replaceable:
\begin{lstlisting}
from sklearn.preprocessing import StandardScaler, MinMaxScaler, RobustScaler

# 1. Standardize (z-score)
sc = StandardScaler().fit(X_train)
X_train_z = sc.transform(X_train)
X_test_z  = sc.transform(X_test)         # NEVER refit on test

# 2. Min-max to [0, 1]
mm = MinMaxScaler().fit(X_train)
X_train_mm = mm.transform(X_train)

# 3. Robust (median / IQR)
rs = RobustScaler().fit(X_train)
X_train_rs = rs.transform(X_train)
\end{lstlisting}
  \vfill
  \footnotesize The \texttt{fit} call learns $\mu, \sigma$ (or min/max, or median/IQR) from \emph{training data only}. The \texttt{transform} call applies the same statistics to both train and test. Pipelines (later) automate this.
\end{frame}

\begin{frame}{When does scaling matter?}
  \begin{center}\small
    \begin{tabular}{p{6.5cm} l p{5cm}}
      \textbf{Model family} & \textbf{Scale?} & \textbf{Why} \\
      \hline
      Linear regression with GD (L02) & yes & step size needs it \\
      Linear regression with normal eq.\ (L01b) & weakly & numerical conditioning \\
      Ridge / Lasso & yes & penalty depends on coeff.\ magnitudes \\
      k-NN, k-means, SVM-RBF & yes & distances dominated by big-range features \\
      Logistic regression with GD & yes & same as linear+GD \\
      PCA & yes & variance dominated by big-range features \\
      Decision trees, random forests, GBMs & \textbf{no} & only use thresholds \\
      Gaussian Naive Bayes & no & per-feature mean/var anyway \\
    \end{tabular}
  \end{center}
  \vfill
  Trees only look at thresholds (\texttt{if area > 60 ...}) - rescaling doesn't change which split is best. Distance-based and gradient-based methods care a lot.
\end{frame}

\section{Cross-cutting: leakage and pipelines}

\begin{frame}{The leakage rule (the single most important slide)}
  Every preprocessor has a \emph{state} learned from data: a mean, a list of categories, a min/max. The rule:
  \vfill
  \begin{center}
    \large
    \textbf{Fit on \textcolor{armblue}{train}. Transform on \textcolor{armred}{train} and \textcolor{armred}{test}.}\\
    Never fit on test. Never fit on train+test combined.
  \end{center}
  \vfill
  Examples of how leakage hides:
  \begin{itemize}
    \item Scaling on full data before train/test split $\Rightarrow$ test mean leaks into train.
    \item Mean-imputing on full data $\Rightarrow$ same.
    \item Target encoding on full data $\Rightarrow$ test target leaks into a feature.
    \item One-hot on full data $\Rightarrow$ usually OK, but unseen test categories will crash unless \texttt{handle\_unknown="ignore"}.
  \end{itemize}
  \vfill
  Effect: inflated CV scores during dev, ugly surprises in production.
\end{frame}

\begin{frame}[fragile]{sklearn Pipeline + ColumnTransformer (the right way)}
  Wrap the chain so leakage is structurally impossible. Conceptually:
  \vskip 4pt
  \begin{center}
    \begin{tikzpicture}[
      node distance=0.35cm and 0.55cm,

\begin{frame}{Recap}
  \begin{itemize}
    \item \textbf{Missing data:} detect the pattern, prefer median/mode for a baseline, use KNN / iterative when you need accuracy, add an indicator column when missingness itself is informative. Drop duplicates after a quick sanity check. Treat outliers as signals to investigate, not noise to delete.
    \item \textbf{Categorical encoding:} one-hot for low cardinality (drop one column if you have an intercept - the dummy trap from L01b). Ordinal only for genuinely ordered categories. Target / frequency / embeddings for high cardinality. Cyclic sin/cos for periodic features like hour-of-day.
    \item \textbf{Scaling:} StandardScaler for Gaussian-ish, MinMax for bounded, Robust for outlier-heavy. Required for GD / kNN / Ridge / Lasso / PCA. Not needed for trees. Log / power transforms come first if the feature is skewed.
    \item \textbf{The one rule:} fit any preprocessor on train data only. Wrap everything in a \texttt{Pipeline} and you can't accidentally violate it.
  \end{itemize}
\end{frame}
