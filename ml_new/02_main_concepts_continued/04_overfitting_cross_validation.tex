\section{The problem: training error lies}

\begin{frame}{Why we need to hold data out}
  \begin{columns}[T]
    \column{0.55\textwidth}
    We fit a model by minimizing empirical risk on the training data (L01, L01b). The fitted model is the one with the lowest training loss.
    \vskip 4pt
    \textbf{Problem:} the training loss is also the thing we used to choose the model. We're \emph{grading the model on the questions we wrote the answer key from}.
    \vskip 4pt
    We need an \textbf{independent estimate} of how the model will do on data it hasn't seen.
    \vskip 4pt
    The picture every ML student should have in mind:

\begin{frame}{Bias-variance tradeoff}
  These two failure modes have formal names you already met in stat lectures (\texttt{03\_stat}): \textbf{bias} and \textbf{variance}, with the identity
  \[
    \mathrm{MSE}(\hat\theta) \;=\; \mathrm{Bias}^2(\hat\theta) \;+\; \mathrm{Var}(\hat\theta)
  \]
  \vfill
  \begin{center}
    \begin{tikzpicture}

        \addlegendentry{Bias$^2$ (underfit)}

        \addlegendentry{Variance (overfit)}
        \addplot[armblue, domain=0.5:9.5, line width=2.5pt] {3.5*exp(-0.3*x) + 0.1*x^1.3 + 0.3};
        \addlegendentry{MSE (total error)}
        \draw[dashed, thick, paramgreen] (axis cs:3.8, 0) -- (axis cs:3.8, 4.3);
        \node[font=\scriptsize\bfseries, paramgreen] at (axis cs:3.8, 4.55) {sweet spot};
        \node[font=\scriptsize] at (axis cs:1.4, 4.0) {too simple};
        \node[font=\scriptsize] at (axis cs:8.2, 4.0) {too complex};

  \vfill

\end{frame}

\begin{frame}{Bias-variance is everywhere in ML}
  The only model class you've met so far is \textbf{polynomial regression}, where the complexity knob is the degree $d$:
  \begin{itemize}\small
    \item Too simple: $d = 1$ (a line through curved data) -- high bias.
    \item Too complex: $d = 20$ (wiggly, like Demo 1) -- high variance.
    \item Sweet spot: somewhere in between, found by validation.
  \end{itemize}
  \vfill
  Every model class you'll meet later has its own ``complexity knob'' and the same U-shape applies:
  \vskip 4pt
  \begin{center}\small
  \begin{tabular}{ll}
    \textbf{Model (upcoming chapter)} & \textbf{Complexity knob} \\
    \hline
    Polynomial regression (now) & degree $d$ \\
    Ridge / Lasso (next chapter) & penalty strength $\lambda$ \\
    KNN (classification chapter) & number of neighbors $k$ \\
    Decision trees (later) & maximum depth \\
    Neural networks (much later) & width / depth, dropout \\
    \hline
  \end{tabular}
  \end{center}
  \vfill
  \fcolorbox{paramgreen}{paramgreen!8}{\parbox{0.97\linewidth}{\small
    \textbf{Key insight.} Different knobs, same shape. This is why we need \textbf{held-out test sets} and \textbf{cross-validation} -- two empirical ways to find the sweet spot, regardless of the model.
  }}
\end{frame}

\begin{frame}{Historical examples}
  \begin{columns}[T]
    \column{0.48\textwidth}
    \textbf{Overfit: Babylonian astrology.}

    For centuries, astrologers built detailed rules connecting planetary positions to historical events:
    \emph{``When Mars is in Taurus and the moon is half full, there will be a great harvest if the newborn prince is left-handed.''}

    They squeezed meaning out of every alignment -- coincidences and noise treated as signal.
    \column{0.48\textwidth}
    \textbf{Underfit: medieval bloodletting.}

    For centuries, European doctors had one rule:
    \emph{``Something's wrong? Drain blood.''}

    Used for headaches, fevers, plague, melancholy alike -- a single too-simple rule for every disease the model couldn't tell apart.
  \end{columns}
  \vfill
  \emph{The art of ML is in the middle: complex enough to match real structure, simple enough not to chase noise.}
\end{frame}

\begin{frame}{The fix: hold out data the model never saw}
  Split the data into two pieces:
  \begin{itemize}
    \item \textbf{Training set} - used to fit the model.
    \item \textbf{Test set} - used \emph{once} at the very end to estimate generalization.
  \end{itemize}
  \vfill
  \begin{center}
    \begin{tikzpicture}
      \fill[armblue!30, draw=armblue, thick] (0,0) rectangle (8,0.8);
      \fill[armred!30, draw=armred, thick] (8,0) rectangle (10,0.8);
      \node[armblue] at (4, 0.4) {\small training (80\%)};
      \node[armred] at (9, 0.4) {\small test (20\%)};
      \draw[<->, gray] (0, -0.2) -- (10, -0.2);
      \node[gray, font=\scriptsize] at (5, -0.5) {full dataset, $n$ rows};
    \end{tikzpicture}
  \end{center}
  \vfill
  Typical splits: 80/20, 70/30 for small data, 90/10 for large data.
  \vfill
  \textbf{Tradeoff:} smaller test set $\Rightarrow$ more data to train on, but noisier estimate. Bigger test set $\Rightarrow$ honest estimate, but less training data. The right ratio depends on how much data you have.
\end{frame}

\begin{frame}{Parameters vs.\ hyperparameters}
  Before we go further, one piece of vocabulary that the rest of this lecture (and the rest of the course) leans on:
  \vfill
  \begin{columns}[T]
    \column{0.5\textwidth}
    \textbf{Parameter $\theta$.} A value the model \emph{learns} from data during training.
    \begin{itemize}\small
      \item Linear regression: the coefficients $\theta_0, \theta_1, \ldots$
      \item Polynomial regression: same -- one per basis column.
    \end{itemize}
    Found by minimizing the empirical risk (L01, L01b).
    \column{0.5\textwidth}
    \textbf{Hyperparameter.} A setting \emph{you} choose before fitting. The model can't learn it because changing it would change the model class itself.
    \begin{itemize}\small
      \item Polynomial regression: the degree $d$.
      \item Gradient descent: the learning rate $\alpha$.
      \item Train/test split: \texttt{test\_size}, \texttt{random\_state}.
    \end{itemize}
    \footnotesize \emph{More hyperparameters appear as we meet new model classes in upcoming chapters.}
  \end{columns}
  \vfill
  \fcolorbox{paramgreen}{paramgreen!8}{\parbox{0.96\linewidth}{\small
    \textbf{Why this matters here:} validation exists to help us pick hyperparameters honestly. ``Try a few values, look at the score, pick the best'' is the whole game for the next several frames.
  }}
\end{frame}

\section{Train/test split}

\begin{frame}{Mechanics: sklearn \texttt{train\_test\_split}}
  Call signature:
  \[
    \texttt{train\_test\_split(X, y, test\_size=, random\_state=, shuffle=, stratify=)}
  \]
  Returns: \texttt{X\_train, X\_test, y\_train, y\_test}.
  \vfill
  \begin{center}\small
    \renewcommand{\arraystretch}{1.4}
    \begin{tabular}{l l}
      \textbf{argument} & \textbf{what it controls} \\
      \hline
      \texttt{test\_size=0.2} & fraction held out (80/20 split) \\
      \texttt{random\_state=509} & seed -- same split every run; required for debugging \\
      \texttt{shuffle=True} (default) & randomize row order before split \\
      \texttt{shuffle=False} & keep chronological order -- use for time-series data \\
      \texttt{stratify=y} & preserve class ratio (classification only -- see Section 5) \\
      \hline
    \end{tabular}
  \end{center}
\end{frame}

\begin{frame}{Critical rules}
  Ranked by how often students miss them:
  \begin{enumerate}
    \item \textbf{Don't iterate on the test set.} The most-violated rule. If you make any modeling decision based on test-set behavior, the test set becomes part of training. Look at test \emph{once}, at the end.
    \item \textbf{Split before any preprocessing.} The leakage rule from L01c (\emph{fit on train, transform on train and test}) requires you to have split first. Without this, mean-imputation and scaling silently leak.
    \item \textbf{Group-structured data needs group splits.} If you have 10 photos per person and randomly split, the same person appears in both train and test. The model memorizes faces, not features. Use \texttt{GroupShuffleSplit}.
    \item \textbf{Time series needs chronological splits.} A random shuffle leaks the future into the past. Use \texttt{TimeSeriesSplit} or a single chronological cutoff.
  \end{enumerate}
  \vfill
  \small\emph{What about EDA?} Distribution-level facts (sizes, classes present, ranges) are fine to check on the full data before splitting. The rule is about \emph{modeling decisions} and \emph{performance numbers}, not about awareness.
\end{frame}

\begin{frame}[fragile]{Nuance: when the rows aren't independent}
  Random shuffle assumes every row is exchangeable. Two cases where this is wildly wrong:
  \vfill
  \textbf{1. Multiple records per entity} (patients, users, photos).
  \footnotesize\\
  100 patients $\times$ 5 visits $=$ 500 rows. Random 80/20 puts $\sim 4$ visits of the same patient in train and $\sim 1$ in test $\Rightarrow$ the model memorizes \emph{the patient}, not the disease pattern. Use \texttt{GroupKFold(...).split(X, y, groups=patient\_id)} -- the \texttt{groups=} argument pins all rows of an entity to a single fold.
  \vfill
  \normalsize\textbf{2. Time-ordered data} (forecasting, sensor readings, prices).
  \footnotesize\\
  Random shuffle puts tomorrow's row in training and yesterday's row in the test set -- training on the future to predict the past. Use \texttt{TimeSeriesSplit(n\_splits=5)} (expanding-window CV), or simply a single chronological cutoff: first 80\% train, last 20\% test.
  \vfill
  \scriptsize \fcolorbox{armred}{armred!8}{\parbox{0.96\linewidth}{\textbf{Diagnosis:} (a) no group ID in both train and test, (b) no test timestamp before the latest train timestamp. Violate either $\Rightarrow$ leaking.}}
\end{frame}

  30 noisy samples from a linear truth ($y = 4 + 3x + \text{noise}$). Fit polynomials of degree 1 to 25 on the training split. For each, measure train MSE and test MSE on the test split.

\section{Why train/test isn't enough: validation}

\begin{frame}{The hyperparameter tuning trap}
  After splitting, you try several models or hyperparameter settings. You evaluate each on the test set, then pick the best.
  \vfill
  Common workflow. Let's check whether the reported number is honest.
  \vfill
  Each comparison \emph{uses} the test set. If you pick the configuration with the best test score, you've absorbed the test set into your model-selection process.
  \vfill
  \textbf{Analogy.} You take a practice exam 100 times until you score perfectly. Do you expect to score perfectly on the real exam?
\end{frame}

\begin{frame}{Three sets, three jobs}
  Each split has exactly one purpose. Mixing them is how reported numbers stop predicting reality.
  \vfill
  \begin{center}
    \begin{tikzpicture}[scale=1.1]
      \fill[armblue!30, draw=armblue, thick] (0,0) rectangle (6,0.8);

    \end{tikzpicture}
  \end{center}
  \vfill
  \begin{center}\small
    \renewcommand{\arraystretch}{1.5}
    \begin{tabular}{l l l}
      \textbf{set} & \textbf{purpose} & \textbf{how often you may touch it} \\
      \hline
      \textcolor{armblue}{\textbf{training}}    & fit \textbf{parameters} ($\theta$)         & as often as you want \\

      \textcolor{armred}{\textbf{test}}         & estimate \textbf{true generalization}      & \textbf{exactly once}, at the end \\
      \hline
    \end{tabular}
  \end{center}
  \vfill
  \fcolorbox{armred}{armred!8}{\parbox{0.97\linewidth}{\small
    \textbf{Why?} Every time you look at a score and \emph{decide} something, you've used that data to fit. Training data fits $\theta$ (allowed). Validation data fits hyperparameters (allowed). If you peek at the test set and adjust anything, you've used it to fit too -- and the number you report is no longer an honest estimate.
  }}
\end{frame}

\section{Cross-validation}

\begin{frame}{One validation set is noisy}
  You have 200 Yerevan listings. 60/20/20 split $\Rightarrow$ 40 validation rows.
  \vfill
  Model A scores 0.81 MSE on validation, model B scores 0.79. Is B really better, or is the gap just noise on 40 samples?
  \vfill
  \textbf{Same data, different random split} can flip which model looks better. On a 200-row dataset the validation score has a standard error of several percent.
  \vfill
  \textbf{Solution:} use $k$ different validation sets and average their scores. Averaging $k$ roughly-independent estimates reduces the variance of the mean by a factor of $\approx 1/k$ (CLT-style, math module 20). This is \textbf{$k$-fold cross-validation}.
\end{frame}

\begin{frame}{$k$-fold cross-validation: how it works}
  Split the (non-test) data into $k$ equal chunks called \emph{folds}. For each of $k$ iterations, hold out one fold as validation and train on the rest.
  \vfill
  \begin{center}
    \begin{tikzpicture}[scale=1]
      \foreach \i in {0,...,4} {
        \node[font=\scriptsize, anchor=east] at (-0.1, -\i*0.55) {iter \the\numexpr\i+1};
        \foreach \j in {0,...,4} {
          \ifnum\i=\j
            \fill[armorange!60, draw=armorange, thick] (\j*0.85, -\i*0.55-0.2) rectangle (\j*0.85+0.8, -\i*0.55+0.2);
          \else
            \fill[armblue!25, draw=armblue, thick] (\j*0.85, -\i*0.55-0.2) rectangle (\j*0.85+0.8, -\i*0.55+0.2);
          \fi
        }
      }
      \node[armorange!80!black, font=\scriptsize, anchor=west] at (4.5, -0.0) {validation fold};
      \node[armblue, font=\scriptsize, anchor=west] at (4.5, -0.55) {training folds};
      \node[gray, font=\scriptsize, anchor=west] at (4.5, -1.65) {final score $=$ average};
      \node[gray, font=\scriptsize, anchor=west] at (4.5, -2.15) {of all $k$ validation scores};
      \node[font=\scriptsize\bfseries, armorange!80!black] at (2, 0.4) {$k = 5$};
    \end{tikzpicture}
  \end{center}
  \vfill
  \fcolorbox{armred}{armred!8}{\parbox{0.96\textwidth}{\small
    \textbf{Subtle point:} you train $k$ different models (one per iteration), not one. The CV score estimates how well the \emph{training protocol} generalizes, not any specific model's performance. The final model you ship is usually fit on \emph{all} the training data after CV picks the hyperparameters.
  }}
\end{frame}

\begin{frame}{Choosing $k$ and the cost}
  \begin{columns}[T]
    \column{0.5\textwidth}
    \textbf{Standard choices:}
    \begin{itemize}\small
      \item $k = 5$ - the default. Gives 80/20 train/val per fold (matches standard hold-out), bias is low, compute manageable.
      \item $k = 10$ - lower variance, 2x more compute. Use when you can afford it.
      \item $k = n$ - leave-one-out (LOO). Train $n$ models. Used only when $n$ is tiny; the LOO score has surprisingly high variance because folds are nearly identical.
      \item \texttt{RepeatedKFold(n\_splits=5, n\_repeats=3)} - repeats $k$-fold with different random splits and averages. Lower variance when compute allows.
    \end{itemize}
    \column{0.5\textwidth}
    \textbf{When CV is worth it:}
    \begin{itemize}\small
      \item Small datasets ($n < 10{,}000$) - one validation set is noisy.
      \item Model selection / hyperparameter tuning.
      \item Reporting a confidence interval on the score (mean $\pm$ std over folds).
    \end{itemize}
    \textbf{When a single split is enough:}
    \begin{itemize}\small
      \item Big data ($n > 10^6$) - one split is already low-noise.
      \item Quick prototyping iterations.
      \item Compute is the bottleneck.
    \end{itemize}
  \end{columns}
\end{frame}

\begin{frame}{CV in code}
  Call signature:
  \[
    \texttt{cross\_val\_score(model, X, y, cv=, scoring=)}
  \]
  Returns: array of $k$ scores (one per fold).
  \vfill
  \begin{center}\small
    \renewcommand{\arraystretch}{1.4}
    \begin{tabular}{l p{0.5\linewidth}}
      \textbf{argument} & \textbf{what it controls} \\
      \hline
      \texttt{model} & any sklearn estimator; refit per fold \\
      \texttt{X, y} & pass \emph{train} subsets only -- test stays untouched \\
      \texttt{cv=KFold(...)} & the splitter (see \texttt{n\_splits}, \texttt{shuffle}, \texttt{random\_state} above) \\
      \texttt{scoring="neg\_mean\_squared\_error"} & error metric (note the sign $\ast$) \\
      \hline
    \end{tabular}
  \end{center}
  \vskip 4pt
  \fcolorbox{armred}{armred!8}{\parbox{0.97\linewidth}{\scriptsize
    $\ast$ \textbf{sklearn footgun:} \texttt{cross\_val\_score} \emph{maximizes} by convention, so error metrics are negated. Flip the sign when reporting: \texttt{-scores.mean()}. Always report \texttt{scores.std()} too -- a single number hides instability.
  }}
\end{frame}

\begin{frame}[fragile]{Pipeline + CV: how the leakage rule survives folds}
  CV trains $k$ models. If you fit your preprocessor (scaler, imputer, encoder) on the full dataset \emph{before} the CV loop, you leak into every fold. The clean fix: wrap preprocessing + model in a \texttt{Pipeline}, then pass the \emph{pipeline} (not just the model) to \texttt{cross\_val\_score}.
  \vfill
{\lstset{basicstyle=\ttfamily\scriptsize}
\begin{lstlisting}
pipe = Pipeline([("scaler", StandardScaler()), ("lr", LinearRegression())])
cross_val_score(pipe, X_train, y_train, cv=KFold(...))
\end{lstlisting}}
  \vfill
  \textbf{Why this works.} On each fold, sklearn refits \emph{every} step of the pipeline on that fold's training rows only. The scaler sees fold-train means/stds, never the fold-val rows. Same for OHE categories, imputer medians, target-encoder per-category means.
  \vfill
  \footnotesize \emph{HW2 used \texttt{ColumnTransformer + LinearRegression}; that pipeline drops straight into \texttt{cross\_val\_score}, no change needed.}
\end{frame}

\begin{frame}{Demo: what happens when you leak}
  Toy demo: 100 rows, 1 numeric feature, scale it. Two ways:
  \vfill
  \begin{center}\small
    \renewcommand{\arraystretch}{1.5}
    \begin{tabular}{p{0.45\linewidth} p{0.45\linewidth}}
      \textbf{\textcolor{armred}{WRONG}} -- scaler fit on full data \emph{before} CV
       & \textbf{\textcolor{paramgreen}{RIGHT}} -- scaler inside Pipeline, refit per fold \\
      \hline
      CV MSE $\approx 0.142$ & CV MSE $\approx 0.158$ \\
      \emph{looks $\sim 11\%$ better} & \emph{honest number, the truth} \\
      \hline
    \end{tabular}
  \end{center}
  \vfill
  \footnotesize On this toy dataset the gap is small ($\sim 11\%$). For \texttt{target encoding} or \texttt{KNNImputer} (which use the target or whole-row similarity), the gap can be \textbf{5--10 percentage points of $R^2$}. Same rule, far worse consequence.
  \vfill
  \fcolorbox{armred}{armred!8}{\parbox{0.96\linewidth}{\scriptsize
    \textbf{Diagnosis:} if your CV score looks suspiciously good, check whether \emph{anything} that was fit before the CV loop touched all the rows. Imputers, scalers, encoders, target encoders, PCA -- all common culprits.
  }}
\end{frame}

\begin{frame}[fragile]{Always have a dumb baseline}
  A CV-mean MSE of 0.16 sounds good. \emph{Is it?} Beats predicting the mean every time? Beats predicting the most common class?
  \vfill
{\lstset{basicstyle=\ttfamily\scriptsize}
\begin{lstlisting}
DummyRegressor(strategy="mean")           # always predicts y.mean()
DummyRegressor(strategy="median")
DummyClassifier(strategy="most_frequent") # always predicts the modal class
DummyClassifier(strategy="stratified")    # predicts proportional to class frequencies
\end{lstlisting}}
  \vfill
  Run the dummy through the same \texttt{cross\_val\_score(...)} as your model. The gap is the part of the score you can claim credit for.
  \vfill
  \fcolorbox{paramgreen}{paramgreen!8}{\parbox{0.96\linewidth}{\small
    \textbf{Workflow rule:} fit a dummy baseline first, every project. Then your fancy model has to clear that bar before you tune it. Saves you from celebrating models that aren't doing anything.
  }}
\end{frame}

\section{Stratified CV (and friends)}

\begin{frame}{The imbalanced-data problem}
  Many real classification tasks are imbalanced:
  \begin{itemize}
    \item Credit-card fraud: 0.2\% fraudulent transactions
    \item Disease screening: 1\% positives in screened population
    \item Click-through prediction: 2\% click rate
    \item Apartment rent fraud detection: maybe 5\% suspicious listings
  \end{itemize}
  \vfill
  Random $k$-fold CV draws folds at random. With a 5\% positive class, the positives are scattered uniformly - but \emph{uniformly} doesn't mean \emph{evenly}. Some folds get more, some fewer, sometimes zero.
  \vfill
  Why care? A fold with 0\% positives breaks the protocol in two ways: the model trained without it never sees the rare class, and the score from that fold isn't meaningful.
  \vfill
  Next frame: how often does this actually happen?
\end{frame}

\begin{frame}{Predict-first: 5-fold CV on 5\% positives}
  You have 200 rows of which 10 are positive (5\%). You run vanilla 5-fold CV.
  \vfill
  \textbf{Questions:}
  \begin{itemize}
    \item How many positives does each fold get \emph{on average}?
    \item What's the probability that at least one fold has zero positives?
  \end{itemize}
  \vfill
  Answers:
  \begin{itemize}
    \item \textbf{On average 2 positives per fold} (10 split across 5).
    \item \textbf{Roughly 6\% chance} at least one fold is empty - the 10 positives can cluster by chance. (Each fold's probability of being empty is $\binom{190}{40}/\binom{200}{40} \approx 0.8^{10} \approx 11\%$; with 5 folds, $\approx 1 - (1 - 0.11)^5 \approx 44\%$ if independent, but folds are negatively correlated $\Rightarrow$ true probability is $\approx 6\%$.) With smaller imbalance (1\% positives, $n=200$) it's nearly guaranteed.
  \end{itemize}
  \vfill
  Solution: split each class separately, so every fold ends up with its share of the minority class. \textbf{Stratified $k$-fold.}
\end{frame}

\begin{frame}{Stratified $k$-fold: vanilla vs stratified}
  Same data, two CV strategies. Each row is one fold's positive count (out of 10).
  \vfill
  \begin{columns}[T]
    \column{0.5\textwidth}

          \fill[armblue!25, draw=armblue, thick] (0, -\i*0.45-0.15) rectangle (4, -\i*0.45+0.15);
          \pgfmathsetmacro{\redw}{\cnt*0.18}
          \ifnum\cnt>0
            \fill[armred!60, draw=armred, thick] (0, -\i*0.45-0.15) rectangle (\redw, -\i*0.45+0.15);

          \fi
        }
      \end{tikzpicture}
    \end{center}

  Stratified $k$-fold splits each class separately. Every fold receives its share. No more empty-positive folds.

    \textbf{For classification: default to \texttt{StratifiedKFold}.} There's almost no reason to use plain \texttt{KFold} on classification data. The sklearn helpers (\texttt{cross\_val\_score}, \texttt{GridSearchCV}) auto-stratify if \texttt{y} is categorical, but be explicit.
  }}
\end{frame}

\begin{frame}[fragile]{Stratified CV in code}
  \footnotesize \emph{Preview of Ch.\ 2 (classification). We use a generic ``classifier'' here; specifics like Logistic Regression, F1, and ROC-AUC come later.}
  \vskip 2pt
\begin{lstlisting}
from sklearn.model_selection import StratifiedKFold, cross_val_score

skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=509)
scores = cross_val_score(
    classifier,           # any classification model
    X_train, y_train,
    cv=skf,
    scoring="accuracy",   # or a metric that handles imbalance better (Ch.2)
)
print(f"score = {scores.mean():.3f} +/- {scores.std():.3f}")
\end{lstlisting}
  \vfill
  \begin{itemize}\small
    \item Use \texttt{StratifiedKFold} \emph{by default} for classification. Almost no reason to use plain \texttt{KFold}.
    \item In Ch.\ 2 we'll see why \texttt{accuracy} is the wrong metric on imbalanced data (a 1\% positive class gets 99\% accuracy from the constant predictor).
  \end{itemize}
\end{frame}

\section{Nested CV}

\begin{frame}{The problem: tuning + evaluating with the same CV}
  The standard three-way split has clean roles: train for parameters, validation for hyperparameters, test for the final number.
  \vfill
  But what if our dataset is small and we already need CV for tuning? Naive workflow:
  \begin{enumerate}\small
    \item Run 5-fold CV across polynomial degrees $d \in \{1, 2, 3, 5, 9\}$.
    \item Pick $d^*$ with the best CV score.
    \item Report that CV score as the model's quality.
  \end{enumerate}
  \vfill
  \fcolorbox{armred}{armred!8}{\parbox{0.96\linewidth}{\small
    \textbf{The bug:} step 3 reports a score that step 2 \emph{used to pick the winner}. Whichever $d$ happened to do well on these specific folds gets chosen -- and we report its score. Same data fitted the choice and graded the choice. Optimism bias: typically 3--10 percentage points.
  }}
  \vfill
  \footnotesize \emph{We need a second layer of CV: one for tuning, one for evaluation. That's nested CV.}
\end{frame}

\begin{frame}{Nested CV: how it works}
  Two CV loops, one inside the other.
  \vfill
  \begin{center}
    \begin{tikzpicture}[scale=0.92]
      % Outer loop: 5 horizontal bars (each row = 1 outer fold split)
      \foreach \k in {0,1,2,3,4} {
        \pgfmathtruncatemacro{\y}{4-\k}
        \foreach \i in {0,1,2,3,4} {
          \ifnum\i=\k
            \fill[armred!60, draw=armred, thick] (\i*1.6, \y*0.6) rectangle (\i*1.6+1.5, \y*0.6+0.5);
          \else
            \fill[armblue!25, draw=armblue!70, thick] (\i*1.6, \y*0.6) rectangle (\i*1.6+1.5, \y*0.6+0.5);
          \fi
        }
        \node[font=\scriptsize, anchor=west] at (8.3, \y*0.6+0.25) {outer fold \k};
      }
      \node[armblue, font=\scriptsize] at (3, 3.5) {outer-train (used inside)};
      \node[armred, font=\scriptsize] at (3, -0.3) {outer-test (one slice, opened once)};
      % Zoom-in: inner CV on a single outer fold
      \draw[->, thick, gray] (10.5, 1.5) -- (12, 1.5);
      \begin{scope}[xshift=12.5cm, yshift=-0.7cm]
        \node[font=\scriptsize\bfseries] at (1.5, 3.8) {inner CV};
        \node[font=\scriptsize] at (1.5, 3.4) {(picks $d$)};
        \foreach \k in {0,1,2} {
          \pgfmathtruncatemacro{\y}{2-\k}
          \foreach \i in {0,1,2} {
            \ifnum\i=\k
              \fill[armorange!60, draw=armorange, thick] (\i*1+0.1, \y*0.55) rectangle (\i*1+0.9, \y*0.55+0.4);
            \else
              \fill[armblue!25, draw=armblue!70, thick] (\i*1+0.1, \y*0.55) rectangle (\i*1+0.9, \y*0.55+0.4);
            \fi
          }
        }
        \node[armorange!80!black, font=\tiny] at (1.5, -0.3) {inner-val};
      \end{scope}
    \end{tikzpicture}
  \end{center}
  \vfill
  \small For \emph{each} outer fold: run a small inner CV on the outer-train to pick the best $d$. Then refit on the whole outer-train and score on the outer-test. The $K$ outer-test scores are the honest estimate.
\end{frame}

\begin{frame}{Nested CV: pattern + when to use}
  Pseudocode:
\begin{center}\small
\fbox{\parbox{0.9\linewidth}{\ttfamily
for tr\_idx, te\_idx in OuterKFold.split(X):          \hfill \textcolor{gray}{\rmfamily\# outer: evaluation}\\
\quad best\_d = inner\_cv\_pick\_degree(X[tr\_idx], y[tr\_idx])  \hfill \textcolor{gray}{\rmfamily\# inner: tuning}\\
\quad score = fit\_and\_score(X[tr\_idx], y[tr\_idx],\\
\qquad\qquad\qquad\qquad\quad X[te\_idx], y[te\_idx], degree=best\_d)
}}
\end{center}
  \vfill
  \textbf{Key properties.}
  \begin{itemize}\small
    \item The outer-test slice is touched \textbf{once} per outer fold (just like a hold-out test set).
    \item Picking $d$ uses only the outer-train -- the outer-test never influenced which $d$ was chosen.
    \item Different outer folds may pick \emph{different} $d^*$ -- that's fine. We're not trying to ship one specific $d$; we're estimating how a \emph{tuning procedure} generalizes.
  \end{itemize}
  \vfill
  \fcolorbox{armred}{armred!8}{\parbox{0.96\linewidth}{\scriptsize
    \textbf{When to use:} any time the number you report is the score of a \emph{tuned} model. The gap between optimism-biased single-CV and honest nested CV can be 3--10 percentage points -- enough to flip ``best in class'' to ``worse than baseline''.
  }}
\end{frame}

\begin{frame}{Recap}
  \begin{itemize}\small
    \item \textbf{Training error lies.} Always evaluate on data the model didn't see.
    \item \textbf{Three sets, three jobs.} Train fits $\theta$; validation fits hyperparameters; test is the final number, touched exactly once.
    \item \textbf{Cross-validation.} Rotates validation through $k$ folds. Default $k=5$.
    \item \textbf{Stratified CV.} Default for classification. \textbf{Group / Time splits.} For non-independent rows.
    \item \textbf{Nested CV.} Honest evaluation when you also tune.
    \item \textbf{Three diagnostic curves.} Validation curve (best HP), learning curve (would more data help?), training curve (when to stop).
  \end{itemize}
  \vfill
  \fcolorbox{paramgreen}{paramgreen!8}{\parbox{0.97\linewidth}{\small
    \textbf{The workflow this lecture supports:}
    \begin{enumerate}\small
      \item \textbf{First fight underfitting.} Pick a model class rich enough to capture the pattern (more features, higher polynomial degree, etc.) -- until train error is low.
      \item \textbf{Then fight overfitting.} Use validation to detect when the model is now memorizing noise, and use \textbf{regularization} (Ridge, Lasso -- \emph{next chapter}) to push complexity back down without losing capacity.
    \end{enumerate}
  }}
  \vfill
  \emph{Coming next: bias-variance decomposition (L01d2), then regularization (L03) -- the principled tool for the overfitting half of the workflow above.}
\end{frame}
